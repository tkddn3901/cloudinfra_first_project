## 🛠️ 테라폼(Terraform) 인프라 구축 및 최적화 기여 요약

본 프로젝트에서 **Infrastructure as Code (IaC)**를 기반으로 버스 혼잡도 데이터와 연동되는 도심지별 가상 네트워크 및 확장형 인프라를 설계하고, 프로젝트의 핵심 파일 구조를 모듈화하여 배포 및 디버깅 전반을 전담했습니다.

### 1. 지역별·역할별 멀티 파일 구조 설계 및 가상 네트워크 구축
* **네트워크 및 라우팅 인프라 코딩 (`network.tf`, `route53.tf`):** 프로젝트의 근간이 되는 VPC, 서브넷 구조를 설계하고 도메인 기반의 트래픽 라우팅을 제어하기 위해 Route 53 호스팅 영역 및 레코드를 효율적으로 코드화했습니다.
* **지역별 컴퓨팅 자원 격리 및 다중화:** 버스 혼잡도 발생 지역의 독립성을 보장하고 트래픽을 효율적으로 격리·수용하기 위해 연산 자원을 멀티 파일 구조로 분리하여 설계했습니다.
  * `compute-gangnam.tf` / `compute-default-gangnam.tf` : 강남권역 인프라 구성
  * `compute-eunpyeong.tf` / `compute-default-eunpyeong.tf` : 은평권역 인프라 구성
* **운영 및 상태 저장 자원 통합 (`compute-mgmt.tf`, `compute-db.tf`):** 인프라 전체를 관리하는 제어용 MGMT 서버와 핵심 데이터를 보관하는 데이터베이스 서버 환경을 독립된 컴포넌트로 구성했습니다.

### 2. 공급자(AWS Provider) 규칙 기반 오토스케일링 트러블슈팅 및 라우팅 최적화
* **StepScaling 및 웜업(Warmup) 메커니즘 교정[cite: 1]:**
  * 각 지역별 인프라가 실시간 혼잡도 트래픽 유입에 맞춰 민첩하게 확장되도록 `aws_autoscaling_policy`를 조율했습니다[cite: 1].
  * 디버깅 과정에서 `StepScaling` 정책 유형 하에 잘못 사용되고 있던 `cooldown = 300` 속성을 감지하고, AWS Provider 규칙에 일치하는 **`estimated_instance_warmup`**으로 전면 교정하여 정책 충돌 오류를 해결하고 확장 민감도를 확보했습니다[cite: 1].
* **로드 밸런싱 인프라 연동 (`loadbalancer.tf`):** 유입되는 버스 이용객 트래픽을 지역별 컴퓨팅 인스턴스 레이어로 안정적으로 분산하기 위해, Application Load Balancer(ALB)와 타겟 그룹(Target Group) 간의 연동 규칙을 구현했습니다.

### 3. 로컬 상태 파일 관리 및 환경 변수 모듈화 (`variables.tf`)
* 인프라의 모든 가변적인 속성(AMI ID, 인스턴스 타입, 지역별 CIDR 블록 등)을 `variables.tf`에 집약하여 코드의 재사용성을 높였습니다.
* 원격 백엔드 대신 로컬 백엔드 환경에서 인프라 상태(`terraform.tfstate`)를 직접 관리하며, 리소스 간 복잡한 참조 관계로 인한 생성·삭제 순서 꼬임(의존성Violation) 문제를 코드 레벨에서 상호 참조 정합성을 맞추어 해결했습니다.
